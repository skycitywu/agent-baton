# AgentBaton — 详细使用指南

**用途**：AgentBaton 是企业团队使用 Claude Code 等 AI Coding 工具进行软件开发协作的一套实践方法。本指南面向人类团队成员，详细介绍各模式下的完整工作流。
**Skill 安装**：见仓库根 [README.md](../README.md)。本指南假设 skill 已安装。

---

## 一句话说明

> 每个团队成员启动 Claude Code 后，告诉 AI "你是什么角色"，AI 就会自动读取框架文件，知道该做什么、怎么做、做完后更新什么。

---

## 核心概念（2 分钟理解）

### 4 个 AI 角色

| 角色 | 谁来操作 | 做什么 |
|------|---------|--------|
| **BA**（业务分析师） | 产品经理 | 把模糊需求变成结构化的 PRD + HTML 原型 |
| **Arch**（架构师） | 开发负责人 | 设计系统架构，把需求拆成开发任务 |
| **Dev**（开发者） | 开发负责人/成员 | 先写测试、再写代码，测试通过才算完成 |
| **TechLead**（技术负责人） | 开发负责人 | 审查架构设计 + 审查代码质量 |

### 3 个关键文件

| 文件 | 作用 | 类比 |
|------|------|------|
| `CLAUDE.md` | 告诉 AI 项目的技术栈和规范 | 项目"身份证" |
| `config/PROJECT_STATUS.md` | AI 的工作日志，每次开工读、收工写 | AI 的"日记本" |
| `config/TECH_STATUS.md` | TechLead 审查意见 ↔ Dev 修复记录 | 团队"便签墙" |

## 项目启动方式（三选一）

根据你的项目情况，选择一种启动方式：

| 方式 | 适用场景 | 从哪里开始 |
|------|---------|-----------|
| **A. 全新项目** | 从零到一的新项目 | 第 0 步 → 第 1-4 步 |
| **B. 接入已有项目** | 已有一期代码（人工开发），要做二期 | 第 0 步 → 基线重建 → 第 1-4 步 |
| **C. 迭代变更** | 已用本框架开发过，用户提出优化/变更 | PM 追加变更描述 → 开发负责人选路径 |

## 方式 A：全新项目完整步骤

### 第 0 步：初始化项目（开发负责人，5 分钟）

```bash
# 创建项目目录并进入
mkdir my-project && cd my-project

# 启动 Claude Code
claude
```

然后在 Claude Code 中输入：
```
用 AgentBaton 初始化新项目
```

Skill 会自动创建完整目录结构。接下来：
- 编辑 `CLAUDE.md`，填写技术栈和规范（「项目阶段」保持默认"全新开发"）
- 初始化 Git：`git init && git add -A && git commit -m "init: 项目初始化"`

### 第 1 步：需求阶段（产品经理，约 1-2 小时）

1. **手写** `docs/INIT_REQ.md` — 用自己的话描述需求，不用完美
2. **打开 Claude Code**，输入：

```
你现在是 BA（业务分析师）角色，请读取 config/roles/BA.md 并按其中第一步完整了解背景。
准备好后告诉我你理解的当前状态和接下来要做什么。
```

3. **和 BA 对话**，回答它的澄清问题
4. **产出**：`docs/PRD.md` + `docs/prototypes/*.html`
5. 用浏览器打开 HTML 原型给客户确认
6. `git commit`

### 第 2 步：架构阶段（开发负责人，约 1-2 小时）

1. **打开 Claude Code**，输入：

```
你现在是 Arch（架构师）角色，请读取 config/roles/ARCH.md 并按其中第一步完整了解背景。
准备好后告诉我你理解的当前状态和接下来要做什么。
```

2. **审核产出**：`docs/ARCH.md` + `docs/TASKS.md`
3. **启动 TechLead 审查架构**：

```
你现在是 TechLead（技术负责人）角色，请读取 config/roles/TECHLEAD.md 并按其中第一步完整了解背景。
准备好后告诉我你理解的当前状态，然后我会告诉你本次审查任务。
```

然后说："请对 docs/ARCH.md 进行架构审查"

4. **架构审查通过后，让 Arch 搭建脚手架**（在同一 Arch 会话中继续，或开新会话）：

```
你现在是 Arch（架构师）角色，请读取 config/roles/ARCH.md 并按其中第一步完整了解背景。
准备好后告诉我你理解的当前状态和接下来要做什么。
```

然后说："架构已通过 TechLead 审查，请按 ARCH.md 的设计搭建项目脚手架（骨架代码），确保项目可构建、各模块接口已定义。"

5. **脚手架搭建完成后，启动 TechLead 审查脚手架**：

```
你现在是 TechLead（技术负责人）角色，请读取 config/roles/TECHLEAD.md 并按其中第一步完整了解背景。
准备好后告诉我你理解的当前状态，然后我会告诉你本次审查任务。
```

然后说："请对脚手架骨架代码进行审查，重点检查：接口签名是否与 ARCH.md 一致、模块边界是否清晰、项目是否可构建。"

6. `git commit`

### 第 3 步：开发阶段（开发负责人/成员，按任务循环）

每个任务的工作流：

1. **打开 Claude Code**，输入：

```
你现在是 Dev（开发者）角色，请读取 config/roles/DEV.md 并按其中第一步完整了解背景。
准备好后告诉我今天要做什么。
```

2. **Dev Agent 会自动**：读任务 → 写测试 → 写代码 → 跑测试 → 更新进度
3. **任务完成后，启动 TechLead 做 Review**
4. **如有问题**，Dev 读 TECH_STATUS.md 修复 → 重跑测试
5. `git commit`

### 第 4 步：集成验收

1. 本地所有测试通过
2. 打包拷贝内网
3. 产品经理对照 PRD + HTML 原型做端到端测试
4. `git tag vX.Y`

## 方式 B：接入已有项目（二期开发）

> 适用于：已有人工开发的一期代码和部分需求文档，需要做二期开发。

### 第 0 步：初始化项目（开发负责人）

```bash
mkdir my-project && cd my-project
claude
```

在 Claude Code 中输入：
```
用 AgentBaton 初始化二期接入项目
```

Skill 会自动创建结构、生成 `docs/phase1/` 目录，并自动把 `CLAUDE.md` 的「项目阶段」设为"二期接入"。

接下来手动编辑 `CLAUDE.md`：
1. 填写技术栈和规范
2.「一期代码仓库」填写一期代码路径

### 第 0.5 步：基线重建（约 2-4 小时）

这是二期项目独有的步骤——让 AI 逆向分析一期代码，重建出基线文档。

**0.5a 准备材料（产品经理 + 开发负责人）**

1. **产品经理**打开 `docs/phase1/INIT_REQ.md`（skill 已创建此模板），用自己的话填写一期做了什么功能，不需要完美，有大概描述即可
2. **开发负责人**在 `CLAUDE.md` 的「一期代码仓库」字段填写一期代码路径（skill 已预留此字段）

**0.5b BA 逆向重建基线 PRD（产品经理操作）**

打开 Claude Code，输入：

```
你现在是 BA（业务分析师）角色，请读取 config/roles/BA.md 并按其中第一步完整了解背景。
准备好后告诉我你理解的当前状态和接下来要做什么。
```

然后说："本项目是二期接入，请重建一期基线 PRD。我会提供一期的需求描述，请结合一期代码分析。"

- **产出**：`docs/phase1/PRD.md`
- `git commit`
- **停下来**，等待开发负责人操作 Arch

**0.5c Arch 逆向重建基线架构（开发负责人操作）**

打开 Claude Code，输入：

```
你现在是 Arch（架构师）角色，请读取 config/roles/ARCH.md 并按其中第一步完整了解背景。
准备好后告诉我你理解的当前状态和接下来要做什么。
```

然后说："本项目是二期接入，请重建一期基线架构和代码导航。phase1/PRD.md 已完成。"

- **产出**：`docs/phase1/ARCH.md` + `docs/phase1/CODEBASE.md`
- `git commit`
- **停下来**，等待人工校验

**0.5d 人工校验基线文档（开发负责人）**

逐一审阅 phase1/ 下的三个文件，对照一期代码验证关键信息。修正明显错误后 `git commit`。

> 从此刻起，phase1/ 文件冻结，不再修改。

**接下来**：产品经理编写二期 `docs/INIT_REQ.md`，然后**走方式 A 的第 1-4 步**（BA/Arch/Dev/TechLead 会自动读取 phase1/ 基线作为约束）。

## 方式 C：迭代变更流程

> 适用于：已用本框架完成一轮开发交付，用户提出优化或变更需求。
> 前提：PRD.md、ARCH.md、TASKS.md 都是最新的（框架流程保证了这一点）。

> **首次进入迭代阶段**：将 `CLAUDE.md` 中的「当前模式」从 `全新开发` 改为 `迭代优化`，仅此一次。

### 第一步：PM 记录变更需求

产品经理打开 `docs/INIT_REQ.md`，在末尾的「迭代需求」区域追加一个子章节，用自然语言描述变更需求，篇幅不限：

```markdown
### v1.1: 导出功能优化
用户反馈导出 Excel 太慢，希望能异步导出，完成后邮件通知。
另外导出的列需要支持自定义选择，目前是固定的。
```

写完后通知开发负责人。

### 第二步：开发负责人判断走哪条路

开发负责人读完 INIT_REQ.md 新增的变更描述后，**二选一**：

---

#### 路径 A：直接开发（不需要评估）

适用于：bug 修复、UI 调整、文案修改、逻辑简单的小功能——**开发负责人一看就知道怎么做**。

**操作**：启动 Dev Agent，输入启动指令后说：

```
INIT_REQ.md 末尾有一条新的迭代需求（v1.1: xxx），请读取，
然后更新 TASKS.md 追加任务，再按正常 TDD 流程实现。
```

Dev 完成后 → 启动 TechLead Review → 修复（如有）→ 完成。

---

#### 路径 B：先评估再开发（需要需求分析或架构考量）

适用于：新功能模块、涉及多模块联动、可能影响架构的变更——**开发负责人觉得需要先想清楚再动手**。

**步骤 1：PM 启动 BA Agent 评估**

```
你现在是 BA（业务分析师）角色，请读取 config/roles/BA.md 并按其中第一步完整了解背景。
准备好后告诉我你理解的当前状态和接下来要做什么。
```

然后说：
```
INIT_REQ.md 末尾有一条新的迭代需求，请读取并评估对现有 PRD 的影响，
与我对话补充细节，然后更新 PRD.md。
```

BA 更新 PRD.md（功能章节 + 验收标准 + 变更记录）→ git commit → **停下来**。

**步骤 2：开发负责人按需操作后续 Agent**

- **需要架构评估**：启动 Arch Agent，让它读更新后的 PRD.md，更新 ARCH.md 和/或 TASKS.md
- **架构有变更**：启动 TechLead 审查架构变更
- **不需要架构评估**：直接启动 Arch Agent 只更新 TASKS.md，或自己手动追加任务

> 开发负责人自行判断哪些步骤必要，没必要的直接跳过。

**步骤 3：Dev 实现 + TechLead Review（正常开发循环）**

---

> **迭代变更关键原则**：
> - 变更需求写在 `INIT_REQ.md`「迭代需求」区域，这是变更的**原始输入**
> - PRD.md 变更记录由 BA Agent 产出（路径 B）或 Dev Agent 追加（路径 A）
> - 在现有文档上追加/修改，**不新建 PRD/ARCH/TASKS 文件**
> - INIT_REQ.md 的原始需求部分**不修改**，迭代需求只在末尾追加

## 每天的工作流程（日常操作）

```
开工：
  打开 Claude Code → 输入角色启动指令 → AI 自动读取进度文件
  → AI 告诉你"上次做到哪了，今天建议做什么"

工作中：
  和 AI 对话完成工作 → AI 按角色指令执行

收工：
  AI 更新 PROJECT_STATUS.md → git commit → 关闭 Claude Code
```

## 各角色速查卡

### 产品经理

```
你的 Agent：BA
你的工作：
  全新项目：手写 INIT_REQ.md → 启动 BA → 产出 PRD + 原型 → 客户确认
  二期接入：整理一期需求描述 → 启动 BA 重建基线 PRD → 再写二期 INIT_REQ.md
  迭代变更：在 INIT_REQ.md 追加变更描述 → 如需评估启动 BA → 更新 PRD
  验收：对照 PRD + 原型做端到端测试
你不需要：写代码、懂技术、配置环境
```

### 开发负责人

```
你的 Agent：Arch / Dev / TechLead（按阶段切换）
你的工作：
  1. 初始化项目，填写 CLAUDE.md（含项目阶段）
  2. 二期项目额外：操作 Arch 重建基线架构，人工校验基线文档
  3. 操作 Arch Agent 做架构设计、任务拆解和脚手架搭建
  4. 操作 Dev Agent 做核心模块编码
  5. 操作 TechLead Agent 审查架构和代码
  6. 迭代变更时：判断直接开发还是先评估，选择路径 A 或 B
  7. 分配任务给开发成员
你是关键人：架构决策、技术审查、变更规模判断、最终质量把关
```

### 开发成员

```
你的 Agent：Dev
你的工作：
  1. 读 TASKS.md 领取任务
  2. 启动 Dev Agent，TDD 方式编码
  3. 二期项目注意：先查 phase1/CODEBASE.md 确认可复用的一期代码
  4. 测试通过后等待 Review
  5. 按 TECH_STATUS.md 修复问题
你不需要：做架构决策、做需求分析
```

## 常见问题

**Q：AI 能记住昨天做了什么吗？**
A：不能，AI 没有跨会话记忆。但每次开工时 AI 会读 `PROJECT_STATUS.md`，里面有完整的工作记录，所以它能"恢复记忆"。**收工前一定要让 AI 更新这个文件。**

**Q：产品经理不懂技术，能操作 Claude Code 吗？**
A：能。只需要复制启动指令粘贴到 Claude Code 中，然后用自然语言和 AI 对话即可。BA Agent 会引导你完成所有工作。

**Q：开发负责人要在 Arch/Dev/TechLead 之间频繁切换？**
A：是的，但每次切换只需要关闭当前会话、开新会话、粘贴对应启动指令。实际上这和日常工作中你在"设计""编码""Review"之间切换是一样的，只是现在有 AI 帮你执行。

**Q：多个开发成员同时开发不同任务会冲突吗？**
A：不会，每个人操作自己的 Claude Code 会话，开发不同的任务。通过 Git 管理代码合并。TASKS.md 中每个任务标注了负责人，避免重复工作。

**Q：内网测试发现 bug 怎么办？**
A：在 `config/PROJECT_STATUS.md` 中记录 bug 为阻塞项，然后启动 Dev Agent 修复。修复后重新打包拷贝内网验证。

**Q：需求变更怎么办？**
A：PM 在 INIT_REQ.md 末尾追加变更描述，然后开发负责人判断：能直接开发就走路径 A（Dev 直接做），需要评估就走路径 B（BA → Arch → Dev）。详见「方式 C：迭代变更流程」。

**Q：二期项目，AI 怎么知道哪些是一期的、哪些是二期的？**
A：通过文件分离。`docs/phase1/` 是一期基线（只读），`docs/PRD.md`、`docs/ARCH.md` 是二期文档。AI 读 CLAUDE.md 时会知道项目阶段，自动区分。

**Q：一期还在并行维护，二期开发会冲突吗？**
A：建议一期维护和二期开发使用 Git 分支隔离。`docs/phase1/` 基线文件是冻结快照，不影响一期代码的实际维护。

## 文件清单（完整列表）

用 AgentBaton 初始化后，项目结构如下：

```
my-project/
├── CLAUDE.md                       # AI 项目说明书
├── config/
│   ├── TEAM_GUIDE.md               # 协作框架（Agent 读）
│   ├── roles/
│   │   ├── BA.md                   # BA 角色指令
│   │   ├── ARCH.md                 # Arch 角色指令
│   │   ├── DEV.md                  # Dev 角色指令
│   │   └── TECHLEAD.md             # TechLead 角色指令
│   ├── PROJECT_STATUS.md           # 项目进度日志
│   └── TECH_STATUS.md              # 技术交接中枢
└── docs/
    ├── INIT_REQ.md                 # 原始需求（产品经理手写）
    ├── PRD.md                      # PRD（BA 产出）
    ├── ARCH.md                     # 架构文档（Arch 产出）
    ├── TASKS.md                    # 任务清单（Arch 产出）
    ├── prototypes/                 # HTML 原型目录
    └── phase1/                     # 仅二期接入项目
        ├── INIT_REQ.md             # 一期功能描述（产品经理手写）
        ├── PRD.md                  # 一期基线 PRD（逆向重建，只读）
        ├── ARCH.md                 # 一期基线架构（逆向重建，只读）
        └── CODEBASE.md             # 一期代码导航（逆向重建，只读）
```
