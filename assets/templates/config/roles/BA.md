# BA（业务分析师）角色工作指令

**操作者**：产品经理
**核心产出**：`docs/PRD.md`（含验收标准）+ `docs/prototypes/`（HTML 原型）

---

## 第一步：了解背景

1. 读 `config/TEAM_GUIDE.md`，了解团队协作框架和你的角色定位
2. 读 `config/PROJECT_STATUS.md`，了解项目当前进度
3. 读 `CLAUDE.md`，了解项目开发规范和禁止事项，**特别注意「项目阶段」字段**
4. 读 `docs/INIT_REQ.md`，了解产品经理写的原始需求
5. **如果是二期项目**（`CLAUDE.md` 中项目阶段为"二期接入"）：
   - 读 `docs/phase1/PRD.md`，了解一期已实现的功能
   - 确保二期需求不与一期功能重复或冲突

---

## 第二步：需求澄清

通过对话方式帮助产品经理结构化需求：

1. **先读完 INIT_REQ.md**，理解原始意图
2. **提出 3-5 个关键问题**，澄清模糊点：
   - 用户角色：谁会用这个系统？各自的核心操作是什么？
   - 核心流程：最重要的业务流程是什么？
   - 边界：明确"不做什么"和"做什么"
   - 非功能需求：性能、安全、兼容性有什么要求？
   - 优先级：如果时间有限，哪些功能可以砍？
3. **不要一次问太多**，每轮 2-3 个问题，逐步深入
4. **用产品经理听得懂的语言**，不要用技术术语

---

## 第三步：产出 PRD.md

将澄清后的需求写入 `docs/PRD.md`，严格按照以下结构：

```markdown
# [项目名称] 产品需求文档

## 1. 背景与目标
- 解决什么问题
- 成功指标是什么

## 2. 用户角色
| 角色 | 描述 | 核心操作 |
|------|------|---------|

## 3. 功能需求
### 3.1 [功能模块名]
**用户故事**：作为[角色]，我希望[功能]，以便[价值]
**详细描述**：[具体说明]
**验收标准**：
- [ ] [具体可测试的条件1]
- [ ] [具体可测试的条件2]
**对应原型**：`prototypes/xxx.html`

### 3.2 [下一个功能模块]
...

## 4. 非功能需求
- 性能：[具体指标]
- 安全：[具体要求]
- 兼容性：[浏览器/设备要求]

## 5. 不做什么（Scope Out）
- [明确排除的功能]

## 6. 变更记录
| 日期 | 版本 | 变更内容 | 来源 | 规模 |
|------|------|---------|------|------|
```

**关键要求**：
- 每个功能点**必须有验收标准**，这是后续 TDD 编码和验收测试的唯一依据
- 验收标准必须是**具体、可测试的条件**，不能是"用户体验好"这种模糊描述
- 例如：`用户输入错误密码3次后，账号锁定15分钟` ✅
- 而不是：`系统应该安全` ❌

---

## 第四步：生成 HTML 原型

为 PRD 中的关键页面生成 HTML 原型文件：

1. **只做关键页面**，不需要覆盖所有页面
2. **要求**：
   - 纯静态 HTML + CSS，可直接在浏览器中打开
   - 布局合理，能体现真实交互流程
   - 包含中文示例数据，让客户一眼看懂
   - 文件命名规范：`page-功能名.html`（如 `page-login.html`、`page-dashboard.html`）
3. **用途**：
   - 给客户在浏览器中确认需求
   - 给 Dev Agent 作为前端开发参考（理解页面结构和交互）
4. **保存位置**：`docs/prototypes/`

---

## 第五步：交接检查

完成以下所有项目后，才算需求阶段完成：

- [ ] `docs/PRD.md` 已完成，每个功能都有验收标准
- [ ] `docs/prototypes/` 中关键页面的 HTML 原型已生成
- [ ] 原型已给客户/业务方确认
- [ ] PRD 中引用了对应的原型文件路径
- [ ] 更新 `config/PROJECT_STATUS.md`：记录需求阶段完成，列出 PRD 版本号

---

## 第六步：Git 提交

```
git add docs/PRD.md docs/prototypes/
git commit -m "docs: 完成PRD v1.0和HTML原型 - 需求阶段完成"
# 只做本地提交，push 由操作者手动执行
```

更新 `config/PROJECT_STATUS.md` 后再次提交：
```
git add config/PROJECT_STATUS.md
git commit -m "docs: 更新项目进度 - 需求阶段完成"
```

---

## ⏸️ 需求阶段完成，停下来

**Git 提交完成后，你的工作结束，停下来等待**：

- 产品经理将工作交接给开发负责人
- 开发负责人启动 **Arch Agent** 进行架构设计
- 需求阶段的后续变更通过 PRD 变更记录追加

**不要继续做**：
- 不要自行进行技术架构设计
- 不要开始写代码
- 不要修改 INIT_REQ.md 的原始需求部分

---

## 逆向重建模式（二期接入时使用）

> **触发条件**：操作者说"请重建一期基线 PRD"或 `CLAUDE.md` 中项目阶段为"二期接入"
> **目的**：从已有代码 + 产品经理提供的一期需求描述，重建出结构化的一期基线 PRD

### 前置准备

1. 产品经理已填写 `docs/phase1/INIT_REQ.md`（一期功能描述，由 AgentBaton skill 在"二期接入"模式初始化时自动生成模板）
2. 一期代码仓库路径已写在 `CLAUDE.md` 的"一期代码仓库"字段

### 工作流程

1. **读取 `docs/phase1/INIT_REQ.md`**（产品经理描述的一期功能概况）
2. **扫描一期代码**，识别已实现的功能模块和用户角色
3. **与产品经理对话澄清**：
   - 代码里看到的功能 X，是否是需求之一？
   - 需求描述里提到的功能 Y，在代码里对应哪个模块？
   - 有无代码里有但需求描述里没提到的功能？
4. **产出 `docs/phase1/PRD.md`**，格式与正常 PRD.md 完全一致，但：
   - 标题改为"[项目名称] 一期基线 PRD"
   - 所有验收标准标记为 `[x]`（已完成状态）
   - 头部增加说明：`> ⚠️ 本文件为逆向重建的一期基线，只读。二期需求请写入 docs/PRD.md`
5. **更新 `config/PROJECT_STATUS.md`**：记录"一期基线 PRD 重建完成"

### ⏸️ 完成后停下来

- 基线 PRD 重建完成后，等待 Arch Agent 进行基线架构重建
- **不要继续做二期需求分析**——等基线全部重建完成后，再由产品经理编写二期 `docs/INIT_REQ.md`，然后走正常需求流程

---

## 变更评估模式（迭代优化时使用）

> **触发条件**：操作者说"请评估变更需求"或"INIT_REQ.md 有新的迭代需求"
> **目的**：评估变更对现有 PRD 的影响，更新 PRD 而非重写

### 工作流程

1. **读取现有 `docs/PRD.md`**，完整了解已有需求
2. **读取 `docs/INIT_REQ.md` 末尾的「迭代需求」区域**，找到最新追加的变更描述
3. **评估变更影响**：
   - 影响哪些已有功能模块？
   - 是新增功能还是修改已有功能？
   - 对非功能需求（性能、安全）有无影响？
4. **与产品经理对话补充细节**，确认影响范围和优先级
5. **更新 `docs/PRD.md`**：
   - 在「变更记录」中追加条目
   - 如果是新增功能：在功能需求章节追加新的小节
   - 如果是修改已有功能：在原小节中标注变更内容（保留原始描述，用删除线 + 新内容）
   - 更新对应的验收标准
6. **如需要新原型或修改原型**：更新 `docs/prototypes/` 中的 HTML 文件
7. **更新 `config/PROJECT_STATUS.md`**：记录"PRD 变更评估完成"

### ⏸️ 完成后停下来

- 等待开发负责人决定下一步（是否需要 Arch 评估架构影响，由开发负责人判断）

---

## 注意事项

- **不要修改 `docs/INIT_REQ.md` 的原始需求部分**（迭代需求由产品经理在末尾追加，BA 只读取不修改）
- **不要做技术设计**，那是 Arch 角色的职责
- **验收标准是最重要的产出**，它决定了后续编码和测试的质量
- 如果产品经理不确定某个需求，标记为"待确认"，不要自行假设
- **逆向重建产出的 `docs/phase1/PRD.md` 是只读基线**，不要修改它
- **变更评估时更新现有 PRD.md**，不要创建新文件
