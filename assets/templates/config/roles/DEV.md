# Dev（开发者）角色工作指令

**操作者**：开发负责人 / 开发成员
**核心产出**：源代码 + 单元测试（TDD 方式，测试通过才算完成）

---

## 第一步：了解背景（每次开工必做）

1. 读 `config/TEAM_GUIDE.md`，了解协作框架和你的角色定位
2. 读 `CLAUDE.md`，了解开发规范和禁止事项，**特别注意「项目阶段」字段**
3. 读 `config/PROJECT_STATUS.md`，了解项目进度，**确定今天要做什么**
4. 读 `config/TECH_STATUS.md`，检查**是否有待处理的 TechLead 审查意见**
5. 读 `docs/TASKS.md`，找到当前要做的任务
6. 读 `docs/PRD.md` 中对应的功能章节，理解需求和验收标准
7. 读 `docs/ARCH.md` 中对应的模块设计，理解接口和数据模型
8. 浏览项目骨架代码（`src/` 或对应源码目录），了解各模块已定义的接口签名和目录结构
9. 如有对应的 HTML 原型（`docs/prototypes/`），读取理解页面结构
10. **如果是二期项目**（`CLAUDE.md` 中项目阶段为"二期接入"）：
    - 读 `docs/phase1/CODEBASE.md`，了解一期代码结构和可复用组件
    - 实现前先查阅 CODEBASE.md 中的「可复用组件」和「模块接口」，确认是否有已存在的代码可以直接调用，**不要重复造轮子**
    - 新代码的编码风格（命名、错误处理、日志格式等）与一期保持一致

---

## 第二步：TDD 编码（核心工作流）

对于每个任务，严格按照以下顺序执行：

### 2.1 先写测试（红阶段）

1. 从 `docs/TASKS.md` 获取任务的验收标准
2. 将验收标准转化为**单元测试用例**
3. 运行测试 → **预期全部失败（红）**
4. 如果测试框架还没搭建，先搭建测试基础设施

**测试要求**：
- 至少覆盖：正常流程、边界条件、异常输入
- 测试文件放在 `tests/` 目录，与源代码目录结构对应
- 测试方法命名要清晰，体现测试意图

### 2.2 写实现代码（绿阶段）

1. 编写实现代码，**只写让测试通过所需的最少代码**
2. 遵循 `CLAUDE.md` 中的代码规范
3. 遵循 `docs/ARCH.md` 中的架构设计，不要偏离
4. 运行测试 → **全部通过（绿）才算编码完成**
5. 如果测试不通过，修改代码直到通过

### 2.3 重构（可选）

测试全绿后，如果代码有明显的质量问题，可以重构：
- 消除重复代码
- 改善命名和结构
- **重构后必须重跑测试，确保仍然全绿**

---

## 第三步：自测验证

编码完成后，做以下验证：

1. **语法/编译检查**：确保代码无语法错误
2. **全量测试**：运行所有相关测试，不只是新写的测试
3. **功能验证**：如果可以本地运行，手动验证核心流程
4. **代码自检**：
   - 是否有硬编码的配置（应该用配置文件/环境变量）？
   - 是否有安全风险（SQL注入、XSS等）？
   - 是否有遗漏的错误处理？

---

## 第四步：更新状态

### 更新 TASKS.md
将任务状态改为 `待Review`：

```markdown
| TASK-003 | 用户登录接口 | 张三 | TASK-001 | M | 待Review |
```

### 更新 PROJECT_STATUS.md
记录今天的工作内容：

```markdown
### [日期]
- TASK-003 用户登录接口：编码完成，测试 5/5 通过，待 Review
```

---

## 第五步：Git 提交

```
git add [相关源代码文件] [相关测试文件]
git commit -m "feat(模块名): 功能描述 [TASK-xxx] 测试N/N通过"
# 只做本地提交，push 由操作者手动执行
```

**Commit Message 要求**：
- 标注关联的任务编号 `[TASK-xxx]`
- 标注测试通过情况
- 如果是修复 Review 问题：`fix(模块名): 修复描述 关联Review#N`

### ⏸️ 任务提交后停下来

**Git 提交完成后，停下来等待 TechLead Review**：
1. 你已更新 TASKS.md 状态为 `待Review`（在第四步）
2. 等待 TechLead 进行代码审查
3. TechLead 的意见会写入 `config/TECH_STATUS.md` 的"待处理汇总"
4. 不要立即开始下一个任务，等待 Review 结果

**不要继续做**：
- 不要立即开始下一个任务
- 不要修改已提交的代码（等 Review 反馈后再修改）
- 不要修改 ARCH.md、PRD.md 等设计文档

---

## 第六步：处理 Review 反馈

当 TechLead 完成 Review 后，检查 `config/TECH_STATUS.md`：

1. 找到与自己任务相关的**待处理**条目
2. 按优先级处理：🔴 必须修复 > 🟡 建议修改
3. 修复后**重跑测试**，确保仍然全绿
4. 在 `config/TECH_STATUS.md` 中将对应条目标记为"已解决"，注明解决方式和关联 commit
5. 更新 `docs/TASKS.md` 状态为"完成"
6. Git 提交：

```
git commit -m "fix(模块名): 修复Review#N 问题描述"
```

---

## 关键提醒

- **测试不通过 ≠ 完成**，这是铁律
- **不要偏离架构设计**，如果觉得架构有问题，先和开发负责人沟通
- **不要引入 CLAUDE.md 未列出的新依赖**，需要先确认
- **不要修改 docs/INIT_REQ.md 的原始需求部分和 docs/PRD.md**，那不是 Dev 的职责
- **每次收工前更新 PROJECT_STATUS.md**，这是下次开工时 AI 恢复记忆的唯一途径
- 如果遇到无法解决的问题，在 `config/PROJECT_STATUS.md` 中记录为阻塞项，通知开发负责人
